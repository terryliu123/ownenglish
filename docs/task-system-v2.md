# 任务系统设计文档 v2

## 1. 文档目标

本文档用于统一当前系统中的任务设计逻辑，覆盖以下场景：

- 课前准备
- 课堂模式
- 学习包
- 学生作答与结果页
- 教师分析

当前版本的核心目标：

1. 拆开“任务模式”和“任务类型”
2. 把现有系统从“功能可用”整理成“结构可扩展”
3. 为后续新增任务类型和轻竞技形态预留统一底座

关联文档：

- `docs/gameplay-design.md`
  - 用于记录课堂竞技玩法、玩法优先级与后续扩展路线

## 2. 当前范围

当前纳入：

- 客观题任务
- 阅读任务
- AI 导入 / AI 生成
- 课堂模式中的发布、作答、查看明细
- 学习包中的任务承载与学习记录
- 教师分析

当前不纳入：

- 复杂多人竞技
- 独立游戏大厅
- 重动画大屏玩法
- 课后自由约战

## 3. 核心概念

### 3.1 任务模式

任务模式决定“任务如何运行”。

当前定义：

- `objective`
  - 以客观题为主
  - 支持实时作答
  - 支持即时判定
- `reading`
  - 以阅读材料为主
  - 支持纯阅读
  - 支持阅读后回答

后续可扩展：

- `expression`
- `discussion`
- `challenge`

### 3.2 任务类型

任务类型决定“任务的具体内容结构”。

当前已接入：

- `single_choice`
- `multiple_choice`
- `true_false`
- `fill_blank`
- `matching`
- `sorting`
- `image_understanding`
- `error_correction`
- `scenario`
- `reading`

后续规划：

- 材料分析题
- 图表理解题
- 步骤推演题
- 观点判断题
- 表达输出题

### 3.3 任务形态

任务形态决定“同一任务在课堂或课后如何被使用”。

当前形态：

- 普通课堂发布
- 学习包模块
- 课后任务

第二阶段新增：

- 单题双人抢答
- 双人 PK
- 全班挑战

约束：

- 轻竞技是任务形态，不是独立题库
- 轻竞技复用现有任务、答案、判题和分析能力

### 3.4 结果类型

结果不只分对错，统一分为：

- `correct`
- `incorrect`
- `completed`
- `unanswered`

这样阅读任务、纯完成型任务和可判对错任务可以共用一套结果体系。

## 4. 统一任务模型

### 4.1 公共字段

所有任务统一支持：

- `id`
- `mode`
- `type`
- `title`
- `question.text`
- `question.image_url`
- `question.image_caption`
- `countdown_seconds`
- `correct_answer`
- `order`
- `ai_meta`

说明：

- 图片统一为题目级字段，不再出现模块级图片语义
- 新任务类型必须复用公共字段，不允许单独发明新的主流程结构

### 4.2 客观题字段

适用于单选、多选、判断、填空、配对、排序、图片理解、纠错、情境题。

可选字段：

- `question.options`
- `question.blanks`
- `question.pairs`
- `question.randomize_answer_position`

### 4.3 阅读任务字段

适用于阅读模式。

可选字段：

- `question.passage`
- `question.prompt`
- `question.answer_required`
- `correct_answer`

规则：

- `answer_required = false` 时只统计完成，不统计正确率
- `answer_required = true` 且无参考答案时统计作答率
- `answer_required = true` 且有参考答案时可统计正确率

## 5. 课前准备设计

### 5.1 定位

课前准备是内容生产中心，负责：

- 创建任务组
- 选择任务模式
- AI 导入 / AI 生成
- 预览
- 编辑
- 确认就绪

### 5.2 当前创建方式

统一保留三种：

- 手动
- AI 导入
- AI 生成

阅读任务与客观题任务都走相同入口，但根据 `task_mode` 呈现不同编辑字段。

### 5.3 当前支持的任务组规则

- 已创建任务组不能回退到第一步重新创建
- 第二步返回后，不新增题目也可继续进入第三步
- 确认就绪后刷新任务列表
- 所有题型支持题目级图片

## 6. 课堂模式设计

### 6.1 职责

课堂模式只负责运行，不负责复杂制作。

当前职责：

- 发布任务组
- 接收学生提交
- 展示课堂状态
- 查看已完成任务明细
- 查看分析
- 结束课堂

### 6.2 课堂记录

课堂普通任务继续按 `session_id` 归档，避免跨场次串数据。

约束：

- 已完成任务明细按 `session_id` 取数
- 历史列表不展示空会话脏数据
- 结束任务后已完成列表立即可见，无需手动刷新

### 6.3 轻竞技最小产品

第二阶段新增两种课堂形态：

- `duel`
  - 双人 PK
  - 指定两名学生参与
  - 先比答对题数，再比总耗时
- `class_challenge`
  - 全班挑战
  - 全体或指定学生参与
  - 排名按答对题数、用时排序

当前仅允许接入以下任务类型：

- `single_choice`
- `multiple_choice`
- `true_false`
- `matching`
- `sorting`
- `image_understanding`

其中：

- `single_question_duel`
  - 仅允许 `single_choice / true_false / image_understanding / error_correction`
- `duel / class_challenge`
  - 继续沿用当前允许集

明确不接入：

- 阅读题
- 开放题
- 口语题

## 7. 学生端设计

### 7.1 课堂模式

课堂模式学生端特点：

- 当前任务唯一焦点
- 实时提交
- 实时结果
- 适合客观题与轻竞技

### 7.2 学习包

学习包学生端坚持分步推进：

- 详情页
- 模块学习页
- 模块完成页
- 整包完成页

禁止把整包内容堆成长页向下滚动。

### 7.3 复学与记录

- 学生完成学习包后允许再次进入任意模块学习
- 系统保存所有历史提交
- 教师分析默认按“每个学生每个模块最后一次提交”统计

## 8. 教师分析设计

### 8.1 主指标口径

课堂模式和学习包统一采用主指标口径：

- `correctness`
- `completion`
- `response`

当前规则：

- 单选、多选、判断、填空、配对、排序、图片理解、纠错、情境题：
  - 使用 `correctness`
- 纯阅读任务、口语完成型任务：
  - 使用 `completion`
- 需要学生回答但没有参考答案的阅读 / 听力任务：
  - 使用 `response`

### 8.2 学习包分析

学习包模块分析统一返回：

- `metric_mode`
- `primary_label`
- `primary_rate`
- `completion_rate`
- `response_rate`
- `correct_rate`
- `sample_answers`

整包分析统一返回：

- `summary_label`
- `summary_rate`

## 9. AI 设计规则

### 9.1 基本原则

- AI 生成的是草稿，不是最终发布内容
- 学习包和课前准备继续共用同一套 AI 配置
- 轻竞技不新增单独 AI 入口

### 9.2 当前接入

- 课前准备
  - AI 导入
  - AI 生成
- 学习包
  - 模块级 AI 导入
  - 模块级 AI 生成
  - 整包级 AI 生成

## 10. UTF-8 与 i18n 约束

前端约束：

- 所有新增中文文案必须走 i18n
- 不再在新增组件中直接写死中文
- 如原始 JSON 已损坏，优先追加 UTF-8 覆盖资源，不在旧文件上继续堆乱码

后端约束：

- 接口错误信息优先使用英文短语，避免继续引入中文常量

## 12. 后续任务扩展记录

固定优先级：

1. 材料分析题
2. 图表理解题
3. 步骤推演题
4. 观点判断题
5. 表达输出题
6. 课后约战

扩展约束：

- 新类型必须接入任务注册表
- 必须定义 `metric_mode`
- 必须复用共享预览、格式化、判题或完成状态逻辑
- 不允许再次在页面中散写独立 `if / else` 主流程
