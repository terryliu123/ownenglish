# 课堂通信稳定性增强计划 v1

日期：2026-04-09

## 目标

把当前互动课堂通信从“基本可用”提升到“长期稳定可运营”的状态，重点解决：

- 空闲断连和静默断连
- 教师/学生断线后的状态恢复
- 关键消息丢失后的补偿
- 课堂运行态过度依赖内存 room
- 单节点长期运行后的脏状态累积

本计划是两部分内容的合并结果：

- 课堂教学通信健壮性审计报告
- 基于当前代码实现现状的补充建议

## 当前实现现状校正

审计报告里有几项描述已经落后于当前代码，需要先统一口径。

### 已经存在的能力

1. 已有基础心跳
- 客户端通用 WS 已有 `ping`
- 服务端已处理 `ping/pong`

2. 已有基础重连
- 通用客户端已有指数退避
- 已有重连次数上限

3. 已有 pending messages 队列
- 客户端并不是完全无队列
- 当前已有队列和上限控制雏形

4. `broadcast_to_students` 不再是“单个学生断线阻断全班广播”
- 当前发送已做逐个连接异常捕获
- 单个学生断线不会直接阻断后续广播

5. `list_classroom_sessions` 已有分页
- “Session 查询无分页”这一条已过时

### 仍然存在的真实短板

1. 心跳策略没有完全统一
- 白板专用 WS 和通用 WS 逻辑仍分散
- 服务端缺少更明确的 missed-pong 清理策略

2. 运行态恢复仍偏依赖内存 room
- 课堂会话、当前任务、当前挑战、提交计数仍主要挂在内存 room
- 数据库恢复能力不够强

3. 关键消息没有确认和补偿
- 发布任务、提交结果、结束课堂仍主要依赖一次发送成功

4. 长时间运行后的 stale room / stale key / 脏连接清理不足

5. 白板页已修复 token 刷新，但“所有 WS 客户端统一 token 刷新”还没完成

## 合并后的问题判断

以下按“是否真实存在”和“是否值得排期”重新归类。

### A. 真实存在且建议进入排期

1. 学生答题提交 DB 写入与教师端实时确认之间缺少补偿机制
- 提交已入库，但教师端实时确认可能失败
- 需要 room 恢复、事件补偿或 ack 机制兜底

2. 老师断线重连后的状态恢复仍不够稳
- 课堂 active session 能恢复，但 room 运行态恢复仍不够强
- 需要更明确的 DB-backed 快照恢复

3. WebSocket 消息 schema 校验需要继续增强
- 当前已有一部分 schema，但还不够完整

4. 后台任务追踪与失败可观测性不足
- `asyncio.create_task(...)` 相关任务还需要统一追踪、告警和清理

5. stale room / stale connection / stale rate-limit key 清理不足
- 这是长期运行稳定性的实际问题

6. 所有 WS 客户端的 token 刷新策略尚未统一
- 白板页已修
- 通用客户端和其他实时页面仍未完全共用

7. 课堂运行态快照不足
- 这是当前最重要的结构性问题

### B. 方向成立，但原始描述需要修正

1. 心跳保活
- 不是“从零增加心跳”
- 应改为“统一心跳 + missed-pong 清理 + 统一断线判定”

2. 客户端重连策略
- 不是“无指数退避、无上限”
- 实际是“已有指数退避和上限，但没有 jitter，且实现分散”

3. `broadcast_to_students` 单点故障
- 不是“一个断连学生阻断全班广播”
- 真实问题是“断开的连接和残留 student ws 清理仍需加强”

4. rate-limit 时间戳累积
- 不是单个列表无界增长
- 真实问题是 stale key 回收不足

### C. 当前不成立或优先级较低

1. Session 查询无分页
- 当前不成立

2. `get_user_from_token` 错误信息不够细
- 值得优化，但不是当前主阻断

3. 挑战赛状态清理竞态
- 需要单独核查，不能直接按报告原文落地

## 合并后的建议方案评估

### 1. 心跳保活

**结论：值得做，优先级 P0，但定义要改。**

#### 不建议按原提法直接做
- 当前已经有 `ping/pong`
- 不是从零加心跳

#### 应该怎么做
- 统一白板专用 WS 与通用 WS 的心跳策略
- 服务端记录每个连接最近一次 `pong`
- 超过阈值未收到 `pong`：
  - 关闭连接
  - 清理 room 中的 stale ws
- 客户端超过一定时间未收到任何消息：
  - 进入主动重连

#### 价值
- 解决大量学校网络、中间代理导致的静默断连

---

### 2. 消息确认 + 重发

**结论：值得做，优先级 P1，但只限关键消息。**

#### 不建议全量协议化
以下消息不值得做 ack：
- 弹幕
- 氛围特效
- 高频 progress
- 普通 presence 变化

#### 建议只覆盖关键消息
- `new_task_group`
- `submission_received`
- `task_ended`
- `room_closed`

#### 建议策略
- 消息带 `msg_id`
- 接收方回 `ack`
- 未收到 `ack` 时，发送方触发：
  - 一次补发
  - 或转为查 `room_info` / 状态恢复

#### 价值
- 解决“到底是没发出，还是发出了但 UI 没收到”的灰区

---

### 3. 断线重连 + 状态恢复

**结论：最值得优先做，优先级 P0/P1。**

#### 当前已有
- `room_info`
- `LiveSession` 恢复
- 教师/学生基础重连

#### 还缺的
- 断线后依赖 DB/快照的强恢复
- 教师刷新后立即恢复当前课堂运行态
- 学生重连后恢复当前任务/挑战/等待态

#### 价值
- 这是当前通信稳定性的核心，不是附属增强

---

### 4. 服务端消息持久化到 DB

**结论：非常值得做，优先级 P0/P1。**

#### 建议重命名为
**关键事件持久化 + 课堂运行态快照**

#### 需要强化的恢复源
- 当前 `live_session_id`
- 当前任务组
- 当前挑战
- 已完成任务历史
- 提交计数
- 课堂摘要相关计数

#### 价值
- 服务重启、老师刷新、异常断线后可以从 DB 恢复，而不是只靠 room 内存

---

### 5. 多进程 / 多节点支持

**结论：方向正确，但应后置到 P3。**

#### 原因
- 单节点语义和恢复链路还没完全收稳
- 过早上 Redis 只会把不稳定状态复制到更多进程

#### 触发条件
- 真正准备多 worker
- 真正准备多实例部署
- 单节点稳定性已经达标

---

### 6. HTTP 降级通道

**结论：值得做，优先级 P0/P1，但要选择性使用。**

#### 不建议双发所有消息

#### 适合 HTTP 兜底的动作
- 学生提交
- 开始本节课
- 结束本节课

#### 不适合变成主通道的
- 实时任务广播
- 挑战实时进度
- 弹幕和氛围特效

#### 轮询降级建议
- WS 断开时自动进入 2s-3s 轮询
- 只拉 `room_info` / 当前课堂状态
- 恢复连接后退出轮询

## 最终排期建议

### P0：立即进入排期

1. 统一 WS token 刷新策略
- 不只白板页
- 所有实时页面统一走共享 token refresh 逻辑

2. 统一心跳与断线判定
- missed-pong cleanup
- stale ws 清理

3. 课堂运行态快照
- 把当前课堂关键状态做成 DB-backed 恢复源

4. 教师/学生断线后的强恢复
- 刷新、重连、网络切换都能恢复当前课堂

5. 关键动作的 HTTP 兜底
- 学生提交
- 结束本节课

### P1：随后的稳定性增强

6. 关键消息 ack / retry
- 仅限少量关键消息

7. WebSocket schema 校验补齐

8. 后台任务 registry + done callback + 错误可观测

9. stale room / stale key / stale connection 清理机制

10. reconnect jitter

### P2：中期增强

11. 提交广播失败后的补偿机制
- outbox / replay / room_info compensation

12. 课堂消息回放能力
- 轻量 ring buffer 或按 `LiveSessionEvent` 增量恢复

13. token 错误分类细化

14. 挑战链路竞态专项核查

### P3：规模化之前再做

15. Redis Pub/Sub

16. Redis Stream / 专用 WebSocket 网关

## 如果只做一件最值钱的事

选：

**课堂运行态快照 + DB 恢复**

原因：
- 当前多数奇怪通信问题，根本原因不是“没有发出去”，而是“发出后状态恢复和状态一致性不稳”
- 只补 transport，不补恢复，问题会反复出现

3. 优先保证：
- 开始本节课
- 发布任务
- 学生接收
- 学生提交
- 教师同步
- 结束本节课
- 学生退出

4. 非关键实时消息不做过度设计
- 不把弹幕、特效、普通 UI 刷新一起拉入 ack 协议

## 总结

这份合并后的结论是：

- 原审计建议整体方向是对的
- 但部分问题描述已经和当前代码不符
- 真正最该优先做的不是“全面补协议”，而是：
  - 统一 token
  - 统一心跳
  - 做强恢复
  - 给关键动作兜底

一句话：

**先把课堂运行态恢复做强，再补关键消息确认，最后再谈多节点扩展。**
