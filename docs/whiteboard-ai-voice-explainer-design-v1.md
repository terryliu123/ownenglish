# 课堂 AI 语音讲解设计稿 v1

## 1. 目标

在白板模式下，为教师端 AI 助手增加 **语音讲解** 能力，用于基于当前白板内容生成一段可播放的讲解音频，并同步展示文字稿。

本期目标不是做开放式实时语音对话，而是先把“AI 能讲”做稳。

## 2. 产品定位

语音讲解是白板 AI 助手的一个扩展动作，定位为：

- 课堂中的辅助讲解工具
- 由老师主动触发
- 基于当前白板上下文生成
- 输出为 `文字稿 + 音频`

不是：

- 全程监听的语音助手
- 自由开放的语音聊天入口
- 学生端共享播报系统

## 3. 首期范围

### 3.1 本期要做

- 教师端白板 AI 助手增加“语音讲解”入口
- 基于当前白板上下文生成 20-40 秒讲解稿
- 支持将讲解稿转换为语音并播放
- 播放时显示文字稿
- 支持：
  - 重新生成
  - 再次播放
  - 停止播放
  - 复制文字稿

### 3.2 本期不做

- 学生端语音 AI
- 实时双向语音对话
- 麦克风输入
- 语音打断
- 全班同步播报
- 多轮语音会话记忆

## 4. 用户场景

### 场景 1：老师需要一段标准讲解

老师板书完成后，点击“语音讲解”，AI 基于当前内容生成一段适合课堂讲解的音频。

### 场景 2：老师需要换一种更简单的说法

老师觉得当前表达不够口语化，可以让 AI 生成一段更适合学生理解的讲解。

### 场景 3：老师需要快速过渡

老师可以直接播放 AI 讲解，再基于文字稿继续补充自己的说明。

## 5. 交互设计

## 5.1 入口

入口继续挂在白板 AI 助手面板中，新增按钮：

- `语音讲解`

位置建议：

- 白板 AI 面板快捷动作区
- 与“解释当前板书”“总结当前板书”同级

## 5.2 流程

1. 老师点击 `语音讲解`
2. 系统收集当前白板上下文
3. AI 生成讲解稿
4. 系统调用 TTS 生成音频
5. 前端展示：
   - 讲解标题
   - 讲解文字稿
   - 音频播放器
6. 老师可操作：
   - 播放
   - 暂停
   - 重播
   - 重新生成
   - 复制文字稿

## 5.3 交互约束

- 默认单次讲解控制在 `20-40` 秒
- 超过 `60` 秒的讲解稿自动截断
- 讲解期间不自动覆盖老师当前白板操作
- 新一次语音讲解会覆盖当前讲解卡片，不保留长历史

## 6. 上下文来源

首期使用两层上下文：

### Level 1：结构化白板文本

- 文本框
- 标题
- 公式文本
- 白板注释

### Level 2：当前课堂内容

- 当前任务预览内容
- 当前互动题标题
- 当前活动标题

本期不启用：

- 整页截图理解
- 复杂手写识别

## 7. 输出规范

AI 输出分两部分：

### 7.1 讲解稿

结构建议：

- `title`
- `script`
- `estimated_seconds`

要求：

- 语言口语化
- 适合课堂直接播报
- 不要生成“作为 AI”之类表述
- 不要出现过长句群

### 7.2 音频

要求：

- 普通话
- 教学场景适用语速
- 音量平稳
- 单次播放完成后可重新播放

## 9. 技术方案

## 9.1 前端拆分

建议新增：

- `client/src/features/whiteboard-ai-voice/components/VoiceExplainButton.tsx`
- `client/src/features/whiteboard-ai-voice/components/VoiceExplainCard.tsx`
- `client/src/features/whiteboard-ai-voice/hooks/useWhiteboardVoiceExplain.ts`
- `client/src/features/whiteboard-ai-voice/services/voiceExplainService.ts`

白板页本身只负责挂载，不直接实现语音逻辑。

## 9.2 后端拆分

建议新增：

- `server/app/api/v1/whiteboard_ai_voice.py`
- `server/app/services/whiteboard_ai_voice.py`

建议接口：

### `POST /api/v1/whiteboard-ai/voice-explain`

请求体：

```json
{
  "class_id": "xxx",
  "context": {
    "whiteboard_text": [],
    "current_task": null,
    "current_activity": null
  },
  "style": "simple_explain"
}
```

返回体：

```json
{
  "title": "欧姆定律核心解释",
  "script": "欧姆定律描述的是……",
  "estimated_seconds": 28,
  "audio_url": "/api/v1/whiteboard-ai/voice-files/abc123.mp3"
}
```

## 9.3 服务端处理流程

1. 构造白板上下文
2. 调 LLM 生成讲解稿
3. 调 TTS 生成音频
4. 返回文字稿和音频地址

## 10. 状态管理

前端 hook 需要管理：

- `loading`
- `playing`
- `error`
- `currentResult`

`currentResult` 至少包含：

- `title`
- `script`
- `estimatedSeconds`
- `audioUrl`

## 11. 风险与约束

### 风险 1：讲解过长

需要在 prompt 和后处理都限制长度。

### 风险 2：上下文不准

如果白板内容抓取不准，讲解会偏离课堂内容。

### 风险 3：课堂控制权被抢

AI 讲解应是老师主动触发，不应自动播放。

### 风险 4：延迟过高

如果生成和合成耗时过长，课堂体验会变差。

首期目标建议：

- 文字生成 + 语音合成总耗时尽量控制在 `3-6` 秒内

## 12. 分阶段实施

### Phase 1

- 只生成文字讲解稿
- 不做语音

### Phase 2

- 接入 TTS
- 播放音频

### Phase 3

- 增加老师端“语音提问”
- 做半双工语音问答

### Phase 4

- 评估是否需要实时语音对话

## 13. 验收标准

1. 老师在白板 AI 助手中能看到“语音讲解”入口
2. 点击后能基于当前白板内容生成讲解稿
3. 能播放生成后的讲解音频
4. 能复制文字稿
5. 重新生成时能覆盖旧结果
6. 不影响现有白板 AI 文字功能

## 14. 结论

最合理的路线是：

- 先做 `AI 语音讲解`
- 再做老师端半双工 `语音提问`
- 最后再评估实时语音对话

这样技术风险最低，课堂可控性最好，也最符合当前白板 AI 助手的产品定位。
