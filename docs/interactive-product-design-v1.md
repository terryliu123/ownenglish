# 互动课堂产品设计 v1

## 1. 产品定位

当前系统定位明确为：

**面向实体教室与直播课堂的实时互动系统。**

核心专注点只有一个：

**互动。**

我们不做：
- 硬件白板平台
- 大而全 LMS
- 复杂课后体系

我们做的是：
- 老师讲解中的即时互动
- 学生实时参与
- 课堂控场与反馈
- 白板驱动的互动体验

## 2. 竞品差异

### 希沃
希沃更强在：
- 校内硬件入口
- 大屏生态
- 公校场景覆盖

我们不正面竞争硬件课堂。

### ClassIn
ClassIn 更强在：
- 在线互动教室
- 机构管理平台
- 课程、作业、测验、录播、后台体系

我们不走大而全平台路线。

### 我们的差异化方向
我们最合理的方向是：
- 白板驱动
- 平板/学生设备参与
- 强实时
- 轻竞技
- 低操作成本
- 兼容实体教室与直播互动

## 3. 商业判断

### 3.1 当前定价
当前建议保留：

- 免费版：基础体验
- 付费版：`9.9 元 / 月`

这个价格当前可以作为：
- 冷启动价格
- 早期种子用户价格
- 低门槛转化价格

### 3.2 后续预留
不建议长期只依赖 `9.9` 作为唯一价格带。

设计上要保留未来升级空间，例如：
- 9.9 体验版
- 更高阶标准版

当前阶段不需要立刻调价，但必须保留升级空间。

## 4. 产品结构重构

### 4.1 主结构
产品主线固定为：

1. 课前准备
2. 白板模式
3. 互动管理
4. 课堂结果反馈

### 4.2 白板模式与课堂模式职责重构

#### 白板模式
白板模式是：

**课堂互动主入口**

主要负责：
- 白板讲解
- 插入互动
- 发起普通互动
- 发起双人 PK
- 发起单题抢答
- 发起全班挑战
- 实时查看课堂互动过程

白板模式代表“课堂现场”。

#### 互动管理
原课堂模式建议产品上更名为：

**互动管理**

主要负责：
- 待发布任务管理
- 历史任务管理
- 查看明细
- 查看分析
- 互动记录复盘

互动管理代表“任务与结果管理”。

### 4.3 分工结论
后续原则固定为：

- 互动类操作，以白板模式为主
- 任务管理、历史记录、数据分析，以互动管理为主

白板和互动管理不再作为两个平级课堂入口竞争，而是：

- 白板 = 上课入口
- 互动管理 = 管理与复盘入口

## 5. 当前系统保留能力

以下能力继续保留并深化：
- 白板模式
- 课前准备
- 普通课堂任务发布
- 双人 PK
- 单题抢答
- 全班挑战
- 课堂明细与分析
- 会员与支付

## 6. 当前系统不再继续扩的方向

当前明确不进入：
- 学习包回归
- 复杂课后体系
- 复杂长尾题型
- 纯大屏游戏体系落地
- 重后台平台能力扩张

## 7. 对现有系统的修改边界

后续所有改动必须满足：

1. 不影响现有白板/互动主链路运行
2. 不破坏已修复的问题
3. 不引入第二套运行态真相源
4. 优先复用现有内容与实时能力
5. 前后端改动都坚持最小侵入

## 8. 下一阶段重点

下一阶段不再以“加新功能”为主，而是：

1. 稳定性优化
2. 产品化收口
3. 支付联调

### 8.1 稳定性
- 白板、互动管理、学生端继续统一运行态
- 实时反馈继续优化
- 主链路回归固化

### 8.2 产品化
- 发起互动路径统一
- 提交反馈统一
- 结束态统一
- 老师点评路径优化

### 8.3 支付与转化
- 微信支付联调
- 会员权益展示与转化体验优化

## 9. 大屏互动的定位

大屏过程型互动后续可以做，但不应直接并入当前任务提交/分析模型。

当前结论固定为：
- 设备参与型互动是现阶段主线
- 大屏过程型互动后续独立设计

## 10. 决策结论

1. 当前系统适合继续围绕“设备参与型互动”做深。
2. `9.9 元 / 月` 作为当前早期付费切入价可行。
3. 白板模式应成为互动主入口。
4. 课堂模式应调整为“互动管理”，专注任务与结果管理。
5. 下一阶段优先做稳定性和产品化，而不是继续扩边界。

一句话结论：

**白板负责课堂现场，互动管理负责任务与结果管理。**
## 11. 第二阶段主线

第二阶段主线不再继续扩当前平板互动模型，而是独立设计“大屏过程型互动”。

设计基线见：

- [大屏互动系统设计 v1](D:\pcode\ownenglish\docs\bigscreen-interaction-system-design-v1.md)

第二阶段约束固定为：

1. 复用内容素材层
2. 不复用当前任务提交、明细、分析模型
3. 不破坏当前白板 / 互动管理主链路
4. 第一批只做配对、排序、分类归组

## 12. 数字化教具能力

系统中现有的静态 HTML 实验 / 教具，需要升级为独立的数字化教具管理能力。

设计基线见：

- [数字化教具管理设计 v1](D:\pcode\ownenglish\docs\digital-teaching-aids-system-design-v1.md)

这条能力的定位是：

1. 作为老师课堂中的数字化演示资源库
2. 与当前 experiment 兼容，但不再继续散落管理
3. 首批支持约 200 个现有静态 HTML 教具批量导入
4. 后续预留 AI 生成 HTML 教具接入能力
