## 2. 项目背景与机会判断

英语直播教学的主要问题不是“没有直播工具”，而是直播课结束后缺少一个真正可持续的学习闭环。

当前老师常见痛点：

1. 直播课结束后，作业布置和内容整理高度重复
2. 学生课后没有稳定练习环境，完成率不高
3. 老师难以快速知道哪些学生没完成、哪些知识点没掌握
4. 老师的课程内容、班级数据和学生资产难沉淀
5. 直播过程中的课堂互动缺少结构化记录，无法直接反哺下一次教学

当前学生常见痛点：

1. 上完直播课后不知道接下来练什么
2. 练习和课堂内容脱节，容易遗忘
3. 反馈滞后，难以知道自己真正的问题
4. 即使完成老师作业，也缺少持续练习入口

因此，OwnEnglish 的切入点不是重新做一套直播系统，而是做一个：

**面向直播英语老师的课中互动 + 课后学习操作系统**

## 3. 市场与竞品结论

### 3.1 竞品赛道分类

当前市场上的相邻竞品主要分为五类：

1. 教学基础设施型：如 `ClassIn`
2. 私域网校经营型：如 `小鹅通`
3. 自营课程交付型：如 `51Talk`
4. 双边撮合 / 订阅口语型：如 `Cambly`、`Preply`
5. 免费学校工具型：如 `Duolingo for Schools`

### 3.2 竞品的核心特点

#### 教学基础设施型

优势：

1. 课堂管理强
2. 支持作业、测试和数据看板
3. 适合机构级运营

不足：

1. 通用教育属性更强
2. 英语专项体验不够深
3. 独立老师品牌沉淀能力有限

#### 私域网校经营型

优势：

1. 招生、转化、分销能力强
2. 私域经营工具成熟
3. 适合课程经营和商业化

不足：

1. 教学体验偏通用
2. 英语练习链路不够垂直
3. 课堂实时互动和学习反馈不够聚焦英语教学本身

#### 自营课程交付型

优势：

1. 课前、课中、课后闭环意识强
2. 学习路径相对完整
3. 测评和作业体系成熟

不足：

1. 平台掌握老师与学生关系
2. 不适合独立老师沉淀自己的班级和内容资产

#### 双边订阅 / 撮合型

优势：

1. 试课与订阅转化机制成熟
2. 口语练习持续性较强
3. 适合持续复购

不足：

1. 更偏交易平台
2. 班级运营和课后学习闭环较弱
3. 老师端教学管理不是核心

#### 免费教师工具型

优势：

1. 容易拉新
2. 留存和轻量打卡设计成熟
3. 对老师友好

不足：

1. 直播课堂衔接较弱
2. 老师自定义内容和班级教学能力有限

### 3.3 我们的差异化空间

经过竞品对比，最值得切入的位置不是泛教育，也不是直播底层，而是：

**面向直播英语老师的小型教学操作系统**

差异化重点应放在：

1. 课中实时任务和课后学习包的双模式闭环
2. 英语专属练习体验，而不是通用题库体验
3. 老师私域资产归属，包括班级、内容和学生沉淀
4. 移动端优先，适配学生真实使用场景
5. 数据直接服务老师下一次教学，而不只是生成报表

### 3.4 竞品可借鉴点

可以吸收的核心做法：

1. 吸收 `ClassIn` 的课堂管理和数据闭环
2. 吸收 `小鹅通` 的私域经营和老师品牌归属意识
3. 吸收 `51Talk` 的课前课后教学闭环
4. 吸收 `Cambly` 的口语持续练习和课后反馈
5. 吸收 `Duolingo` 的轻量任务和留存机制

## 4. 产品定位

### 4.1 一句话定位

OwnEnglish 是一款服务直播英语老师的教学辅助平台，帮助老师把一节直播课转化为“课中可互动、课后可巩固、数据可追踪”的完整学习闭环。

### 4.2 产品本质

它不是直播平台本身，而是：

1. 老师的教学辅助系统
2. 学生的课后练习与复习空间
3. 平台免费线索内容的补充学习区

### 4.3 产品边界

第一阶段明确不做：

1. 自建直播音视频底层
2. 复杂社区和排行社交
3. 多老师课程 marketplace
4. 重 AI 全自动课程系统
5. 机构级复杂后台

### 4.4 类目策略与扩展方向

当前设计出来的底层范式，本质上并不只适用于英语，而是适用于更广义的直播教学增强场景：

`课中实时互动 + 课后学习包 + 学习反馈 + 下次教学调整`

但在市场切入和产品聚焦上，第一阶段仍然明确以英语作为抓手。

#### 当前对外定位

1. 对外仍定义为英语学习产品
2. 第一核心用户仍是直播英语老师
3. 第一阶段内容与能力建设优先服务英语场景

#### 对内架构原则

1. 底层班级、任务、学习包、数据分析能力尽量做通用
2. 学科差异主要沉淀在内容层、题型层和能力层
3. 不把系统数据模型写死为“只支持英语”

#### 类目分层建议

平台未来可按三层分类来设计：

##### 第一层：学习类目

1. 英语
2. 其他语言
3. 其他学习类目

##### 第二层：学习目标

1. 综合提升
2. 口语表达
3. 听力训练
4. 阅读理解
5. 写作表达
6. 考试备考
7. 场景沟通

##### 第三层：内容形态

1. 课堂实时任务
2. 课后学习包
3. 录音 / 跟读练习
4. 微练习
5. 公开课 / 免费内容

#### 阶段性扩展路径

建议按以下顺序扩展：

1. 阶段 1：先做英语直播教学闭环
2. 阶段 2：在英语内部扩展更多学习类型与内容方向
3. 阶段 3：扩展到其他语言类目
4. 阶段 4：再评估是否扩展到更泛的直播学习类目

#### 英语作为第一抓手的原因

1. 英语老师的需求最集中
2. 英语场景更容易做出垂直差异化体验
3. 英语的口语、听力、句型、跟读等能力链路更适合第一版验证
4. 先垂直切入，比一开始做泛类目平台更容易跑通产品价值

一句话原则：

**战略上先垂直英语，架构上预留多类目扩展能力。**

## 5. 产品价值主张

### 5.1 对老师的价值

1. 上完直播课后，1 分钟内能发布一套高质量课后学习包
2. 课堂中可以快速发起实时任务，马上知道学生是否听懂
3. 教学数据能直接帮助下一次备课和复讲
4. 内容、班级和学生关系沉淀在自己的长期阵地中

### 5.2 对学生的价值

1. 上完课立刻知道今天该练什么
2. 可以在手机或平板上快速完成老师任务
3. 能即时获得结果和反馈
4. 做完老师作业后，还有免费线索内容继续提升

## 6. 目标用户

### 6.1 核心用户 1：直播英语老师

典型特征：

1. 有固定直播授课场景
2. 有班级与学生群体
3. 需要课中互动、课后作业、结果跟踪
4. 希望沉淀自己的私域教学资产

核心诉求：

1. 快速建班和发作业
2. 直播中能看学生实时反馈
3. 课后有结构化练习工具
4. 能看到谁掉队、哪里薄弱

### 6.2 核心用户 2：跟老师上课的学生

典型特征：

1. 主要通过手机学习，部分场景使用平板
2. 上课时需要快速响应课堂任务
3. 课后需要明确的练习安排与反馈
4. 容易被复杂任务流打断，需要轻量化体验

核心诉求：

1. 进入班级简单
2. 课中答题快
3. 课后任务清晰
4. 反馈及时
5. 有可继续提升的补充练习

## 7. 产品模式

平台明确拆分为两种任务模式，不能混为一个系统入口。

### 7.1 课堂实时任务

目标：

帮助老师在直播课堂中快速判断学生是否听懂。

特点：

1. 老师实时发起
2. 学生立即作答
3. 结果实时汇总到老师端
4. 服务于课堂节奏控制和复讲判断

### 7.2 课后学习包

目标：

帮助学生在课后完成系统复习和巩固。

特点：

1. 老师课后发布
2. 学生按自己的节奏完成
3. 支持词汇、句型、听力、阅读、口语等模块
4. 服务于课后学习闭环和结果沉淀

### 7.3 免费学习区

目标：

承担平台拉新、续学和补练功能。

特点：

1. 不打断老师主线任务
2. 可作为未入班用户体验入口
3. 可作为已入班学生的补充练习区

## 8. 商业模式与推广模式

### 8.1 第一阶段商业模式

建议优先采用“三层结构，轻重分明”的商业设计。

#### 老师端收费

1. 按老师账号订阅
2. 按班级数或高级功能分层收费
3. 课堂实时任务、数据分析、学习包模板等可以作为高级能力

#### 学生端弱收费

1. 老师指定任务基础免费
2. 免费学习区开放部分内容
3. 进阶专项训练、会员内容可以后续收费

#### 平台内容合作

1. 后续可开放老师内容合作
2. 但第一阶段不以内容交易平台为主

### 8.2 推广路径

优先推广对象应为老师，而不是泛学生。

推广模式建议：

1. 通过老师试用切入班级
2. 通过免费线索学习包给学生体验
3. 通过老师班级分享和私域传播带来学生
4. 通过老师品牌主页与公开练习做自然扩散

### 8.3 核心增长策略

1. 免费教师工具拉新
2. 免费内容区承接学生
3. 课堂任务和课后作业形成使用习惯
4. 数据反馈提升老师留存

## 9. 核心设计原则

1. 老师核心动作尽量控制在 3 步以内
2. 学生打开系统后，只看到当前最重要的任务
3. 课堂模式强调速度和判断，不强调复杂内容
4. 课后模式强调完整练习和结果反馈
5. 数据必须能指导老师的下一次教学
6. 学生端必须同时支持手机和平板，但不能只是简单等比放大
7. 视觉上要简洁、专业、温和，不能像通用后台模板

## 13. 关键用户流程

### 13.1 老师课堂实时流程

1. 老师进入课堂模式
2. 先配置一组实时题队列
3. 为每题设置题型、题干、选项、倒计时和结果策略
4. 点击发布整组任务到当前班级
5. 学生收到提醒并进入当前题作答
6. 老师实时查看人数、选项分布和正确率
7. 老师决定复讲、重发、跳题或切换下一题
8. 整组课堂互动结果被保存到班级数据中

### 13.2 老师课后流程

1. 老师进入控制台
2. 新建课后学习包
3. 选择课程与班级
4. 系统推荐练习模块与题目结构
5. 老师编辑模块、题目顺序、时长和截止时间
6. 发布整包学习任务
7. 次日查看完成率和薄弱点

### 13.3 学生课堂实时流程

1. 学生上课时收到任务提醒
2. 进入实时答题页
3. 在倒计时内提交答案
4. 看到已提交状态
5. 等待老师讲解或下一题

### 13.4 学生课后流程

3. 完成词汇、听力、句型、阅读练习
4. 完成口语录音并提交
5. 查看学习报告
6. 进入免费区继续练习

## 15. 学生端设备策略

学生端必须同时支持两种移动设备形态：

1. 手机
2. 平板

### 15.1 手机端设计要求

1. 单列布局
2. 强主任务结构
3. 适合拇指点击的大按钮和大选项
4. 适合课堂实时快速响应
5. 适合 5 到 10 分钟碎片练习

### 15.2 平板端设计要求

1. 允许双栏或更舒展布局
2. 左侧可承载任务概览，右侧承载主内容
3. 适合课堂实时答题和更长时段复习
4. 信息密度更高，但保持与手机端一致的品牌体验

### 15.3 设计原则

1. 平板端不是手机端的等比放大
2. 两端必须保持一致的操作路径
3. 同一任务链路在手机和平板都必须成立

## 23. 学习包学生端交互补充说明

### 23.1 设计原则

学习包在学生端采用“分步推进式学习流程”，不采用“长页面向下滚动完成整包内容”的交互方式。

原因：

1. 学习包属于课后非实时任务，与课堂模式的实时答题节奏不同。
2. 学生端主要在手机和平板上使用，长页滚动会降低专注度和完成率。
3. 分步推进更容易建立进度感、完成感和任务边界。
4. 更适合支持断点续学、顺序解锁、必做模块和选做模块。
5. 更利于后续做模块级反馈、整包结果页和下一步推荐。

### 23.2 与课堂模式的区别

课堂模式：

1. 实时互动。
2. 以当前题为唯一焦点。
3. 倒计时强约束。
4. 强调即时响应和课堂反馈。

学习包：

1. 非实时学习。
2. 以模块和步骤推进为主。
3. 支持暂停、退出、继续学习。
4. 强调完成课后任务和巩固学习内容。

结论：

学习包学生端不能复用课堂模式的页面结构，也不能将整包内容平铺在一个页面中让学生持续下拉完成。

### 23.3 推荐交互结构

学生端学习包推荐采用以下流程：

1. 学习包详情页。
2. 学习进行页。
3. 模块完成页。
4. 学习包完成页。

#### 23.3.1 学习包详情页

展示内容：

1. 学习包标题。
2. 学习目标。
3. 预计总时长。
4. 截止时间。
5. 完成规则。
6. 模块数量。
7. 开始学习按钮。

#### 23.3.2 学习进行页

一次只展示“当前步骤”，不展示整包全部内容。

页面结构建议：

1. 顶部进度条。
2. 当前模块标题。
3. 当前模块目标说明。
4. 当前任务项内容。
5. 保存并继续。
6. 返回任务概览。

说明：

1. 若模块内有多个任务项，则逐项推进。
2. 若老师设置顺序规则，则完成当前模块后进入下一模块。
3. 若老师允许自由学习，则学生可返回概览切换模块。

#### 23.3.3 模块完成页

用于模块结束后的短反馈，建议展示：

1. 当前模块已完成。
2. 当前模块完成情况。
3. 是否进入下一模块。
4. 下一模块名称。

#### 23.3.4 学习包完成页

用于整包结束后的总结反馈，建议展示：

1. 学习包完成状态。
2. 完成时长。
3. 模块完成情况。
4. 正确率或完成率。
5. 口语/提交任务状态。
6. 推荐下一步。

### 23.4 推进规则

学习包内部建议采用两层推进：

1. 包级推进：模块 1 -> 模块 2 -> 模块 3。
2. 模块内推进：任务项 1 -> 任务项 2 -> 任务项 3。

默认推荐：

1. 学习包按步骤推进。
2. 一个时刻只聚焦一个当前任务。
3. 支持中途退出和断点续学。

### 23.5 禁止作为默认方案的交互

1. 整包模块与题目在一个页面中全部展开。
2. 学生通过长页面连续下拉完成所有内容。
3. 直接复用课堂实时答题页作为学习包页。

### 23.6 设计结论

学习包学生端应设计为：

分步式、流程式、可断点续学的课后学习体验。
