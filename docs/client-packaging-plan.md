# 客户端打包方案记录

## 目标
当前系统继续以 Web 为主，不立即改造为桌面端或原生端。本文件只记录后续可执行的打包方案，待需要时再实施。

目标拆分：
- 教师端：打包为 Windows 下可安装运行的 `exe`
- 学生端：打包为 Android 平板可安装运行的 `app`

## 当前判断
系统现阶段仍然是典型 Web 应用，核心价值在：
- 白板模式
- 课堂模式
- 实时互动
- WebSocket 状态同步
- 轻竞技

因此后续打包不建议重写原生，而建议采用“Web 主体 + 原生壳”的方式。

## 推荐方案

### 1. 教师端 Windows 可执行程序
推荐方案：`Electron`

原因：
- 与当前 React / Vite 前端兼容最好
- 对白板、拖拽、全屏、投屏、WebSocket 等能力兼容性更稳
- Windows 安装包与 `exe` 生态成熟
- 后续可加自动更新

备选方案：`Tauri`

适用场景：
- 追求更小包体
- 追求更低内存占用

当前不作为第一选择的原因：
- 桌面端生态与踩坑成本相对 Electron 更高
- 第一版更应优先稳定性，而不是极致轻量

教师端结论：
- 第一版优先采用 `Electron`

### 2. 学生端 Android 平板 App
推荐方案：`Capacitor`

原因：
- 可直接复用现有前端页面
- 可产出 Android 安装包
- 比纯 WebView 壳更适合后续接入：
  - 麦克风权限
  - 文件/图片权限
  - 全屏显示
  - 横屏控制
  - 音频上传
  - 前后台恢复

备选方案：纯 Android WebView 壳

适用场景：
- 快速演示
- 临时测试包

不建议作为正式产品方案的原因：
- 后续权限、音频、缓存、实时状态恢复会越来越难维护

当前不建议方案：
- Flutter 重写
- React Native 重写

原因：
- 成本过高
- 会显著拖慢课堂主链路迭代

学生端结论：
- 第一版优先采用 `Capacitor`

## 推荐架构

### 教师端
- 保留现有 Web 版本
- 使用 `Electron` 封装现有教师前端

### 学生端
- 保留现有 Web 版本
- 使用 `Capacitor` 封装现有学生前端，生成 Android 平板 App

该路线的优势：
- 共用同一套业务代码
- 最小化重写成本
- 仍可持续用 Web 方式快速迭代课堂功能
